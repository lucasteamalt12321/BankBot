# Базовая инструкция для ИИ (Code Agent)

## Общие принципы

- Отвечать на русском языке и следовать инструкциям пользователя.
- Использовать пакетный менеджер bun.
- Всегда проверять файлы после внесения изменений на линтинг с помощью biome и автоматически исправлять их.
- Не проверять Markdown-файлы (`*.md`) с помощью Biome.
- Не спрашивать разрешения перед просмотром файлов проекта, а сразу использовать инструменты для просмотра.
- Не предлагать код для нескольких вариантов решения. Автоматически выбирать лучшее на основе мировых практик и применять его автоматически.
- Никогда самостоятельно не изменять что-либо, кроме того что запланировано или попросил пользователь. Предлагать такие действия в конце отчета о выполненной задаче.
- При изменении элементов стиля всегда учитывать текущий вариант темы (темный/светлый) и изменять второй соответствующим образом.
- Не коммитить изменения на Github самостоятельно. Только по команде пользователя (если пользователь просит закоммитить, то можно).

## Система Memory Bank

Ты действуешь как инженер, который не обладает долгосрочной памятью. Твой единственный устойчивый контекст — директория `/memory_bank`. Поддерживай документацию в актуальном состоянии, синхронизируя её с кодом.

### 1. Фундаментальные правила (Policy)

- `docs/README.md` — единственный источник информации о высокоуровневой архитектуре.
- Канонический источник процента выполнения: `memory_bank/projectbrief.md`.
- В `memory_bank/projectbrief.md` обязателен раздел `## Project Deliverables`.
- Создать раздел автоматически, если он отсутствует, при любой задаче связанной с прогрессом, планированием или синхронизацией memory_bank.
- `## Project Deliverables` — канонический список результатов, по которому считается процент выполнения. Не журнал сессии.
- Каждый deliverable обязан иметь: стабильный ID, краткое название, статус (`pending` / `in_progress` / `completed` / `blocked`), вес в процентах.
- Сумма весов всех deliverables = `100`.
- Процент выполнения считается только по `## Project Deliverables`.
- `activeContext.md`, `progress.md`, `Changelog` — операционный контекст и история, не источник процента.
- Каждый новый пункт ТЗ или scope change добавлять в `## Project Deliverables` до или одновременно с реализацией.
- После каждой задачи, влияющей на готовность — обновить статусы deliverables.
- Если deliverables не синхронизированы с кодом — сначала восстановить, потом считать процент. Иначе процент = `нет достоверных данных`.

### 2. Память проекта

- Перед началом любой работы перечитай `/memory_bank`, особенно `activeContext.md` и `progress.md`.
- `/memory_bank` — твоя операционная система. Без актуализации теряешь контекст.
- Любые изменения в архитектуре, маршрутах или модулях отражать в `docs/README.md`.

### 3. Контроль изменений

- `progress.md` всегда содержит `last_checked_commit` (хеш или дата).
- При каждом сеансе сравнивай: `git log <last_checked_commit>...`
- При изменении документации — обновить хеш.

### 4. Структура Memory Bank

Обязательные файлы в `/memory_bank`:

| Файл | Назначение |
|------|-----------|
| `projectbrief.md` | Цели, рамки проекта, Project Deliverables |
| `productContext.md` | Продуктовый контекст, проблемы пользователей |
| `activeContext.md` | Текущий фокус, активные решения и приоритеты |
| `systemPatterns.md` | Архитектура, паттерны, связи подсистем |
| `techContext.md` | Стек, окружение, библиотеки, CI/CD |
| `progress.md` | Статус, Known Issues, Changelog, last_checked_commit |

Дополнительно: `modules/*`, `ui_extension/*`, `other/*`

### 5. Рабочие режимы

**Режим А — Планирование** (крупные/архитектурные/рискованные задачи):
1. Прочитай `productContext.md` и `activeContext.md`.
2. Составь план: шаги, файлы, риски.
3. Не приступай без одобрения пользователя.

Для повседневных правок и точечных фиксов — выполнять сразу.

**Режим Б — Выполнение:**
1. Обнови `activeContext.md` — задача взята в работу.
2. Выполняй последовательно.
3. Обновляй память: CI/CD → `techContext.md`; новый функционал → `docs/README.md`; legacy → `other/` + `progress.md`.
4. Обнови `progress.md` (Changelog) и `last_checked_commit`.

**Режим В — Инициализация** (Memory Bank пуст):
1. Создай файлы-заглушки.
2. Составь карту директорий и точек входа.
3. Запиши в `techContext.md` и `systemPatterns.md`.

### 6. Чек-лист перед завершением сессии

1. Проверил `git status` на неожиданные изменения?
2. Обновил `progress.md` и `activeContext.md`?
3. Синхронизировал `docs/README.md` при изменении архитектуры?
4. Указал новый `last_checked_commit`?

## План разработки проекта (BankBot)

### Фаза 1 — Основа (Приоритет: критический)
1. Вынести все конфиденциальные данные (токены, ID, пароли) в переменные окружения
2. Создать централизованную конфигурацию на базе Pydantic Settings с поддержкой окружений (dev/prod/test)
3. Закрепить версии зависимостей, разделить на `requirements.txt` и `requirements-dev.txt`
4. Исправить все импорты после рефакторинга конфигурации

### Фаза 2 — Архитектура (Приоритет: высокий)
5. Реализовать слой репозиториев для всех операций с БД, убрать прямые SQL запросы
6. Создать service layer, вынести бизнес-логику из handlers
7. Разбить монолитный `bot.py` на модули по группам команд
8. Настроить централизованный middleware для обработки ошибок с уведомлением администратора
9. Реализовать graceful shutdown с корректным закрытием ресурсов
10. Перенести конфигурацию парсинга из JSON файлов в БД, создать ParserRegistry

### Фаза 3 — Безопасность и качество (Приоритет: средний)
11. Добавить блокировки для операций с балансом, реализовать Unit of Work для атомарных транзакций
12. Настроить connection pooling
13. Провести аудит SQL запросов на SQL injection, добавить параметризацию везде где её нет
14. Реализовать систему алиасов пользователей с fallback на имя
15. Настроить DI контейнер зависимостей

### Фаза 4 — Очистка (Приоритет: низкий)
16. Провести аудит неиспользуемого кода (D&D система, система кланов, неиспользуемые таблицы БД)
17. Удалить или задокументировать deprecated функции
18. Объединить дублирующиеся парсеры

### Фаза 5 — Тесты и документация (Приоритет: низкий)
19. Написать E2E тесты для основных сценариев (регистрация, начисление очков, покупки)
20. Добавить тесты на SQL injection, race conditions, graceful shutdown
21. Довести coverage до 80%+
22. Обновить README, создать DEPLOYMENT.md, архитектурные диаграммы
23. Добавить docstrings (Google style) во все публичные методы
