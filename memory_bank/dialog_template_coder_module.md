# Module Plan — Dialog Template Coder

## Priority

**MAX / P0 / выполнить в первую очередь.**

Этот модуль имеет максимальный приоритет относительно текущих pending-задач post-release roadmap (`PR10–PR13`). Перед архитектурной зачисткой, legacy-cleanup и прочими P2-задачами нужно реализовать или спланировать именно этот диалоговый модуль.

## Назначение

Telegram-бот должен поддерживать пошаговый диалоговый кодер текстовых шаблонов. Пользователь отправляет только текстовые шаблоны словами, без цифр. Бот хранит состояние пользователя, подтверждает текущую последовательность, показывает расшифровку и предлагает допустимые следующие шаблоны.

## Основной принцип

- Бот поддерживает состояние для каждого пользователя: `A` и `B`, если они уже выбраны.
- Пользователь не вводит цифры, только текстовые шаблоны, например: `Скоро буду`, `ОК`, `Спасибо, нет`.
- На каждый ввод бот отвечает одним сообщением в формате:

```text
Вы написали «[последовательность]», что значит «[расшифровка на текущий момент]».
Варианты для следующего шаблона:
[шаблон1] → «[расшифровка]»
[шаблон2] → «[расшифровка]»
...
```

- Если последовательность завершена — получены 3 шаблона — бот пишет итог и сбрасывает состояние.
- После сброса пользователь может начать новую последовательность обычным первым шаблоном.

## Формат ответов по шагам

### Шаг 1 — получен только A

Пример:

```text
Вы написали «Скоро буду», что значит «Скоро буду / Иду».
Варианты для второго шаблона (любой из 10):
ОК → «Буду через 5 минут»
Да → «Да, иду»
Спасибо → «Спасибо, скоро буду»
"Спасибо, нет" → «Не приду»
Великолепно → «Буду с отличным настроением»
Спасибо еще раз → «Скоро буду, спасибо за ожидание»
Скоро увидимся → «Скоро буду, потом увидимся»
Скоро буду → «Скоро-скоро буду»
Я занят(а) → «Задержусь, но буду»
Нет → «Не скоро буду»
```

### Шаг 2 — получены A и B

Пример:

```text
Вы написали «Скоро буду → ОК», что значит «Буду через 5 минут».
Теперь вы можете прислать третий шаблон из списка: ОК, Да, Спасибо, Великолепно, Нет.
Варианты:
ОК → «Буду через 5 минут»
Да → «Буду через 10 минут»
Спасибо → «Буду через 15 минут»
Великолепно → «Буду через 2 минуты»
Нет → «Буду через 20 минут»
```

### Шаг 3 — получены A, B и C

Пример:

```text
Вы написали «Скоро буду → ОК → Да», что значит «Буду через 10 минут». Конец последовательности.
Чтобы начать новую, просто напишите новый первый шаблон.
```

После этого состояние сбрасывается.

## Обработка ввода

- Бот приводит ввод к каноническому виду из списка 10 шаблонов.
- Регистр не важен.
- Примеры нормализации:
  - `спасибо нет` → `"Спасибо, нет"`
  - `я занят` → `Я занят(а)`
  - `спасибо еще раз` → `Спасибо еще раз`
- Если пользователь ввёл нераспознанный текст, бот отвечает:

```text
Не понял. Пришлите один из шаблонов: ОК, Да, Спасибо, "Спасибо, нет", Великолепно, Спасибо еще раз, Скоро увидимся, Скоро буду, Я занят(а), Нет.
```

## Ошибки и особые ситуации

- На шаге 2 принимается любой из 10 шаблонов.
- На шаге 3 принимаются только 5 шаблонов: `ОК`, `Да`, `Спасибо`, `Великолепно`, `Нет`.
- Если на шаге 3 введён другой допустимый, но запрещённый для третьего шага шаблон, бот отвечает:

```text
На третьем шаге разрешены только: ОК, Да, Спасибо, Великолепно, Нет. Вы написали «...». Попробуйте ещё раз.
```

- Если пользователь начал новую последовательность, не завершив старую — например, после `A` прислал новый `A` — старую можно сбросить и начать заново. Бот должен написать: `Начинаю новую последовательность. Вы написали ...`.
- `/reset` — сбросить состояние.
- `/help` — вывести описание и список шаблонов.

## Необходимые данные

Разработчику нужны встроенные таблицы:

- 10 шаблонов: текст ↔ цифровой код.
- Таблица одиночных значений: 10 фраз.
- Таблица пар: 100 записей, для каждой пары `(A, B)` — фраза.
- Таблица троек: 500 записей, для каждой тройки `(A, B, C)`, где `C ∈ {1, 2, 3, 5, 10}` — фраза.

Таблицы должны быть перенесены в JSON-словари или прямо в код. Если полные списки на 100 и 500 записей ещё не лежат в репозитории, нужно запросить/добавить их перед реализацией.

### Таблица одиночных значений

Бот показывает эти значения, когда пользователь прислал ровно один шаблон и ещё не прислал второй.

| Код | Шаблон | Значение для первого шага |
|-----|--------|----------------------------|
| 1 | ОК | Всё в порядке / ОК |
| 2 | Да | Да |
| 3 | Спасибо | Спасибо |
| 4 | "Спасибо, нет" | Нет, спасибо |
| 5 | Великолепно | Великолепно / Отлично |
| 6 | Спасибо еще раз | Спасибо ещё раз |
| 7 | Скоро увидимся | До скорой встречи |
| 8 | Скоро буду | Скоро буду / Иду |
| 9 | Я занят(а) | Я занят(а) |
| 10 | Нет | Нет |

```json
  "2": "Да",
  "3": "Спасибо",
  "4": "Нет, спасибо",
  "5": "Великолепно / Отлично",
  "6": "Спасибо ещё раз",
  "7": "До скорой встречи",
  "8": "Скоро буду / Иду",
  "9": "Я занят(а)",
  "10": "Нет"

```

### Правило третьего уровня

Третий шаблон `C` принимает только значения `{1, 2, 3, 5, 10}`:

| C | Шаблон | Семантика модификатора |
|---|--------|-------------------------|
| 1 | ОК | базовое, нейтральное |
| 2 | Да | усиленное, больше, быстрее, радостнее |
| 3 | Спасибо | вежливое, благодарное |
| 5 | Великолепно | максимальное, превосходное |
| 10 | Нет | ослабленное, медленнее, отрицательное |

### Важная заметка по таблице 500 троек

Пользователь предоставил полный список: `100 пар × 5 троек = 500`.
При переносе в JSON нельзя генерировать тексты автоматически по шаблону модификатора: нужно сохранить конкретные фразы из предоставленного списка, потому что у многих троек есть индивидуальные формулировки, например `(8,1,2) → «Буду через 10 минут»`, `(9,8,5) → «Через 30 минут»`, `(10,10,5) → «Ни за что»`.

## Хранение состояний

- Ключ состояния: `chat_id`.
- Поля состояния:
  - `step`: `0`, `1`, `2`;
  - `A`: код от 1 до 10;
  - `B`: код, если `step >= 1`.
- При `step = 2` бот хранит пару `(A, B)` и ждёт третий шаблон.
- Опционально: сброс состояния через 30 минут бездействия.

## Технические детали реализации

- Язык: Python 3.10+.
- Библиотека: `python-telegram-bot` async.
- Использовать `ConversationHandler` с тремя состояниями:
  - `AWAITING_FIRST`
  - `AWAITING_SECOND`
  - `AWAITING_THIRD`
- В `AWAITING_THIRD` проверять, что введённый шаблон входит в `{ОК, Да, Спасибо, Великолепно, Нет}` в каноническом виде.
- Фразы для вариантов на каждом шаге генерируются динамически из таблиц пар/троек:
  - шаг 1: показать все 10 вариантов `(A, B)`;
  - шаг 2: показать 5 вариантов `(A, B, C)`;
  - шаг 3: показать итоговую расшифровку и сбросить состояние.
- Обязательно учесть шаблон `"Спасибо, нет"` с кавычками и запятой для корректного отображения в ответах.

## Команды

- `/start` — приветствие, сброс состояния, показ списка шаблонов.
- `/reset` — сброс состояния, аналогично `/start`.
- `/help` — инструкция и список шаблонов.

## Результат работы

Бот полностью готов к деплою, все таблицы встроены, диалог идёт строго по формату:

```text
Вы написали … → … что значит … Варианты: …
```

## Acceptance Criteria

- Пользователь может пройти последовательность из 3 шаблонов.
- После первого шага бот показывает 10 вариантов второго шаблона.
- После второго шага бот показывает только 5 разрешённых вариантов третьего шаблона.
- После третьего шага бот выводит итог и сбрасывает состояние.
- Нераспознанные тексты получают понятную ошибку со списком допустимых шаблонов.
- `/start`, `/reset`, `/help` работают независимо от текущего состояния.
- Состояния разделены по `chat_id` и не смешиваются между пользователями/чатами.
- Таблицы одиночных значений, 100 пар и 500 троек встроены или подключены из JSON.
