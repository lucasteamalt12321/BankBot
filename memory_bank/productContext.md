# Product Context

## Проблемы пользователей

- Игроки участвуют в нескольких игровых платформах (GD Cards, Shmalala, True Mafia, Bunker RP)
- Каждая платформа имеет свою валюту и систему достижений
- Нет единой системы вознаграждений и прогрессии
- Сложно отслеживать активность на всех платформах одновременно
- На Hugging Face локальная файловая БД ephemeral: без PostgreSQL пользователи, балансы и feedback могли теряться после restart/rebuild
- В группах Telegram длинные ответы и накопленные pending updates могут приводить к таймаутам/flood-control, поэтому нужны короткие режимы ответа
- Часть пользователей управляет ботом с часов, где экран помещает только короткий текст и доступны только фиксированные quick-reply шаблоны
- Администраторам нужен быстрый способ видеть feedback/жалобы, управлять балансами, рассылками и режимами ответа без молчаливых ошибок

## Решение

Telegram-бот BankBot объединяет все игровые платформы в единую экосистему. Главная цель — автоматически парсить игровые сообщения и на этой основе надёжно начислять банковские монеты:
- Единый баланс банковских монет
- Автоматическое начисление очков за активность в разных играх
- Общий магазин товаров и привилегий
- Система достижений, охватывающая все платформы
- Persistent production storage через PostgreSQL/Supabase на Hugging Face; SQLite остаётся local/dev fallback
- Feedback inbox: `/feedback`, `/suggest`, `/complaint`, админский `/feedback_list` и защищённый внешний reader `/feedback`
- Бесплатный локальный AI-lite помощник (`/ai`, `/ask`, `/ai_help`) для навигации по командам и локальной базе знаний
- Режимы ответа `/short`, `/long`, `/watch`; админские `/short_all`, `/long_all`, `/watch_all`
- Watch-mode: управление с часов через 10 шаблонов (`ОК`, `Да`, `Спасибо`, `Спасибо нет`, `Великолепно`, `Спасибо еще раз`, `Скоро увидимся`, `Скоро буду`, `Я занят(а)`, `Нет`)

Банк, админка, PostgreSQL, feedback, режимы ответа и AI-lite — не замена парсинга, а инфраструктурный фундамент вокруг главной функции: парсинга игровой активности. Текущий честный статус: инфраструктура и ручной контур есть, production E2E автоматический парсинг нужно стабилизировать как PARSE01.

## Scope Correction — Webhook-first BankBot

Пользователь уточнил новый продуктовый фокус для стабилизации: production на Hugging Face должен перейти с polling на webhook, чтобы убрать регулярные Telegram `getUpdates` timeout'ы и сделать групповые команды/парсинг предсказуемыми.

После webhook-переноса продуктовый минимум BankBot:
- надёжный приём сообщений из Telegram групп через webhook;
- безопасный парсинг только по реальному reply на сообщение игры;
- единый баланс и история транзакций;
- `/start`, `/user`/`/profile`, `/balance`, `/history`, `/stats`;
- краткий/полный режимы ответа: только `/short` и `/long`;
- базовая админка для балансов и диагностики;
- feedback, если не мешает webhook runtime.

Пользователь также решил убрать/отключить из production runtime то, что сейчас не работает или не нужно для основной цели:
- `/shop`, `/games`, `/dnd`;
- BridgeBot и VK Bot;
- фоновые periodic tasks;
- watch/ADB/ntfy realtime flows;
- небезопасный fallback, где пользователь вручную вставляет текст карты/профиля.

## Целевая аудитория

- Активные игроки в GD Cards, Shmalala, True Mafia, Bunker RP
- Администраторы игровых сообществ
- Организаторы игровых мероприятий
- Пользователи, которые читают и отвечают с часов/коротких уведомлений

## Ключевые метрики успеха

- Количество активных пользователей (> 100)
- Среднее количество транзакций в день (> 500)
- Уровень удержания пользователей (> 70%)
- Количество обрабатываемых игровых сообщений (> 1000/день)
- Стабильность HF runtime: `/health` healthy, polling отвечает на `/ping`, нет повторяющихся runtime errors
- Доля успешных feedback/admin операций без молчаливых ошибок
- Использование режимов `/short`/`/watch` для снижения таймаутов длинных ответов
